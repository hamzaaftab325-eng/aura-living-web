     1‚Üí  1301‚ÜíThese principles are not aspirational. They are enforced by an
     2‚Üí  1302‚ÜíESLint plugin (eslint-plugin-import with import-boundaries
     3‚Üí  1303‚Üírules) that fails the build on violation. A developer who tries to
     4‚Üí  1304‚Üíimport from features/cart/store.ts directly from app/page.tsx
     5‚Üí  1305‚Üíwill see a lint error before they can commit.
     6‚Üí  1306‚Üí5. Design System
     7‚Üí  1307‚ÜíThe Aura Living design system is the single source of truth for
     8‚Üí  1308‚Üíevery visual decision in the application. It is encoded as CSS
     9‚Üí  1309‚Üícustom properties (design tokens) in globals.css, surfaced as
    10‚Üí  1310‚ÜíTailwind theme extensions in tailwind.config.ts, and consumed
    11‚Üí  1311‚Üíexclusively via Tailwind utility classes in components. There are
    12‚Üí  1312‚Üíno inline styles anywhere in the codebase. There are no magic
    13‚Üí  1313‚Üínumbers in components. Every colour, every spacing value, every
    14‚Üí  1314‚Üíradius, every shadow, every animation duration is a token, and
    15‚Üí  1315‚Üíevery token has a name, a value, and a documented purpose. This
    16‚Üí  1316‚Üídiscipline is what allows a designer and a developer to collaborate
    17‚Üí  1317‚Üíwithout friction and what allows the brand to evolve without
    18‚Üí  1318‚Üícascading refactors.
    19‚Üí  1319‚Üí5.1 Color Token System
    20‚Üí  1320‚Üí
    21‚Üí  1321‚Üí===== PAGE 34 =====
    22‚Üí  1322‚Üí
    23‚Üí  1323‚ÜíAura Living uses a four-tier colour token system inspired by the
    24‚Üí  1324‚Üí2026 Tailwind 4 design-token conventions. The four tiers are:
    25‚Üí  1325‚Üíprimitive (the raw colour value, e.g., a specific hex), semantic (the
    26‚Üí  1326‚Üírole the colour plays, e.g., "background" or "foreground"),
    27‚Üí  1327‚Üícomponent (the colour as applied to a specific component, e.g.,
    28‚Üí  1328‚Üí"button-primary-background"), and state (variations for hover,
    29‚Üí  1329‚Üífocus, active, disabled). Primitive tokens are never used directly in
    30‚Üí  1330‚Üícomponents; they are composed into semantic tokens, which are
    31‚Üí  1331‚Üícomposed into component tokens. This indirection means that
    32‚Üí  1332‚Üíchanging the brand gold from #C9A84C to a slightly warmer
    33‚Üí  1333‚Üíshade requires editing one primitive token, not hunting through
    34‚Üí  1334‚Üíevery component.
    35‚Üí  1335‚ÜíThe colour system is anchored by three primary families: gold (the
    36‚Üí  1336‚Üíbrand accent, in five shades), black (the ink and dark-section
    37‚Üí  1337‚Üíbackground, in four shades), and white (the light-section
    38‚Üí  1338‚Üíbackground, in two warm shades). Two supporting families ‚Äî
    39‚Üí  1339‚Üíwarm gray (for muted text and borders) and signal colours
    40‚Üí  1340‚Üí(success green, warning amber, error red, used sparingly for
    41‚Üí  1341‚Üísystem feedback) ‚Äî complete the palette. Every colour in the
    42‚Üí  1342‚Üísystem meets WCAG 2.2 AA contrast against at least one other
    43‚Üí  1343‚Üícolour in the system, and the gold-on-black combination meets
    44‚Üí  1344‚ÜíAAA contrast (7.4:1), making it safe for body text in dark sections.

   162‚Üí  1462‚Üíe-commerce sites. Inter is chosen as the body face because it was
   163‚Üí  1463‚Üídesigned for screen readability at small sizes, has exceptional
   164‚Üí  1464‚Üílegibility, and is neutral enough to let Fraunces carry the brand

   268‚Üí  1568‚Üí5.4 Spacing System
   269‚Üí  1569‚ÜíAura Living uses an 8-pixel base spacing unit, extended with a 4-
   270‚Üí  1570‚Üípixel half-step for fine adjustments (icon padding, tight groups).
   271‚Üí  1571‚ÜíAll spacing in the application ‚Äî padding, margin, gap, position
   272‚Üí  1572‚Üíoffsets ‚Äî is a multiple of 4. This constraint is enforced by the
   273‚Üí  1573‚ÜíTailwind config: only the values in the spacing scale are available
   274‚Üí  1574‚Üías utilities, and arbitrary values via square-bracket syntax are
   275‚Üí  1575‚Üídisabled by an ESLint rule. The result is that every component sits
   276‚Üí  1576‚Üíon the same invisible grid, and the eye perceives a calm rhythm
   277‚Üí  1577‚Üíacross the entire site.
   278‚Üí  1578‚Üí
   279‚Üí  1579‚Üí===== PAGE 39 =====
   280‚Üí  1580‚Üí

   282‚Üí  1582‚Üíthe section level. Section padding (the space between major page
   283‚Üí  1583‚Üísections) is clamp(3rem, 8vw, 6rem) on mobile to desktop, scaling
   284‚Üí  1584‚Üísmoothly rather than stepping at breakpoints. Component-level
   285‚Üí  1585‚Üíspacing stays fixed: a button's internal padding does not change
   286‚Üí  1586‚Üíbetween mobile and desktop, only its container's spacing does.

   410‚Üí  1710‚Üí6. Animation & Motion Strategy
   411‚Üí  1711‚ÜíMotion in Aura Living is not decoration; it is communication.
   412‚Üí  1712‚ÜíEvery animation has a purpose: to draw attention, to provide
   413‚Üí  1713‚Üífeedback, to guide the eye, to smooth a transition, or to delight.
   414‚Üí  1714‚ÜíAnimations without a purpose are removed. This discipline is
   415‚Üí  1715‚Üícritical because animations have a cost ‚Äî they consume main-
   416‚Üí  1716‚Üíthread time, they can cause layout shift, and they can make a site
   417‚Üí  1717‚Üífeel sluggish if mistimed. The Aura Living motion system is
   418‚Üí  1718‚Üíengineered to feel expensive (long, smooth, choreographed) on
   419‚Üí  1719‚Üícapable devices and to feel instant on low-powered devices, with
   420‚Üí  1720‚Üíno middle ground that feels neither polished nor fast.
   421‚Üí  1721‚ÜíThree libraries work together, each with a clearly delineated role.
   422‚Üí  1722‚ÜíGSAP handles scroll-triggered animations (parallax, pinned
   423‚Üí  1723‚Üísections, scrub-linked reveals), long-running timeline-based
   424‚Üí  1724‚Üísequences (the homepage hero entrance, the product page gallery
   425‚Üí  1725‚Üítransitions), and any animation that requires precise timing
   426‚Üí  1726‚Üícontrol across multiple elements. Framer Motion (Motion) handles
   427‚Üí  1727‚Üícomponent-level animations (enter/exit transitions, layout
   428‚Üí  1728‚Üíanimations, gesture-based interactions like drag and hover, and
   429‚Üí  1729‚ÜíAnimatePresence-driven mount/unmount). Lenis provides smooth
   430‚Üí  1730‚Üíscrolling on desktop and (selectively) on mobile, integrating with
   431‚Üí  1731‚Üí
   432‚Üí  1732‚Üí===== PAGE 42 =====
   433‚Üí  1733‚Üí
   434‚Üí  1734‚ÜíGSAP's ScrollTrigger via a one-time raf wiring. There is deliberate
   435‚Üí  1735‚Üíoverlap between what GSAP and Framer Motion can do; the rule
   436‚Üí  1736‚Üíis that GSAP owns anything scroll- or timeline-driven, and Framer
   437‚Üí  1737‚ÜíMotion owns anything component- or gesture-driven.
   438‚Üí  1738‚Üí6.1 Motion Design Philosophy
   439‚Üí  1739‚ÜíAura Living motion follows four principles. First, slow is luxurious:
   440‚Üí  1740‚ÜíAura Living animations are slower than typical web animations
   441‚Üí  1741‚Üí(entrance animations are 800ms to 1200ms, not 300ms to 500ms)
   442‚Üí  1742‚Üíbecause slow motion signals confidence and craft. Second,
   443‚Üí  1743‚Üíchoreography over chaos: when multiple elements animate
   444‚Üí  1744‚Üítogether, they animate in sequence with small staggers (50ms to
   445‚Üí  1745‚Üí120ms apart), never simultaneously, which creates a sense of
   446‚Üí  1746‚Üíchoreography. Third, ease over line: every animation uses a
   447‚Üí  1747‚Üícustom cubic-bezier ease (a slow-in, slow-out curve with a slight
   448‚Üí  1748‚Üíovershoot for organic motion) rather than linear or default ease
   449‚Üí  1749‚Üícurves. Fourth, respect the user: every animation respects the
   450‚Üí  1750‚Üíprefers-reduced-motion media query, falling back to instant state
   451‚Üí  1751‚Üíchanges for users who have requested reduced motion.
   452‚Üí  1752‚ÜíCUSTOM EASE CURVE
   453‚Üí  1753‚ÜíAura Living's signature ease: cubic-bezier(0.22, 1, 0.36, 1). This
   454‚Üí  1754‚Üíis a refined ease-out with no overshoot, derived from the
   455‚Üí  1755‚Üístandard 'easeOutQuint' curve. It feels slower and more
   456‚Üí  1756‚Üídeliberate than the default ease-out, which gives Aura Living
   457‚Üí  1757‚Üímotion its luxurious quality.
   458‚Üí  1758‚Üí6.2 GSAP Usage Patterns
   459‚Üí  1759‚ÜíGSAP is used via the @gsap/react package's useGSAP hook, which
   460‚Üí  1760‚Üíis the SSR-safe drop-in for useLayoutEffect and includes
   461‚Üí  1761‚Üíautomatic cleanup via gsap.context(). Every GSAP animation in
   462‚Üí  1762‚ÜíAura Living lives inside a useGSAP hook, and the hook's scope is

   464‚Üí  1764‚Üícomponents.

   565‚Üí  1865‚Üíscroll event. On desktop, this gives the entire site a buttery,
   566‚Üí  1866‚Üípremium feel. On mobile, the situation is more nuanced: native

   614‚Üí  1914‚Üí6.5 Parallax Implementation
   615‚Üí  1915‚ÜíParallax is the signature motion effect of Aura Living, used on the
   616‚Üí  1916‚Üíhomepage hero, on editorial banners throughout the site, on the
   617‚Üí  1917‚Üílookbook detail pages, and on the product page's lifestyle imagery.
   618‚Üí  1918‚ÜíThe implementation is GPU-accelerated (transform-only, never
   619‚Üí  1919‚Üílayout properties like top or left), passive (no scroll event
   620‚Üí  1920‚Üílisteners, only ScrollTrigger's raf-driven updates), and reduced-
   621‚Üí  1921‚Üímotion-aware (instant fallback). The parallax speed is calibrated
   622‚Üí  1922‚Üíper device: stronger on desktop (where the effect reads as
   623‚Üí  1923‚Üíluxurious) and subtler on mobile (where strong parallax causes
   624‚Üí  1924‚Üímotion sickness and feels gimmicky).
   625‚Üí  1925‚ÜíThree parallax patterns are used. The first is single-layer parallax,
   626‚Üí  1926‚Üíwhere a background image moves slower than the foreground
   627‚Üí  1927‚Üícontent (speed -30 to -50 on desktop, -15 to -25 on mobile). The
   628‚Üí  1928‚Üísecond is multi-layer parallax, where two or three layers move at
   629‚Üí  1929‚Üídifferent speeds to create depth (used on the homepage hero: a
   630‚Üí  1930‚Üíbackground image at -50, a mid-ground product image at -25, and
   631‚Üí  1931‚Üía foreground text overlay at 0). The third is scroll-linked rotation,
   632‚Üí  1932‚Üíwhere an element rotates subtly based on scroll progress (used on
   633‚Üí  1933‚Üíthe Aura Living logo mark in the homepage hero, which rotates 5
   634‚Üí  1934‚Üídegrees as the user scrolls past).
   635‚Üí  1935‚Üí6.6 Reduced Motion Accessibility
   636‚Üí  1936‚ÜíEvery animation in Aura Living respects the prefers-reduced-
   637‚Üí  1937‚Üímotion media query. Users who have enabled this preference in
   638‚Üí  1938‚Üítheir operating system see instant state changes with no motion.
   639‚Üí  1939‚ÜíThis is not optional; it is a WCAG 2.2 AA requirement and a basic
   640‚Üí  1940‚Üírespect for users who experience motion sensitivity, vestibular
   641‚Üí  1941‚Üídisorders, or simply prefer a calmer interface. The
   642‚Üí  1942‚ÜíuseReducedMotion hook (built on top of Framer Motion's hook of
   643‚Üí  1943‚Üíthe same name) is the single point of truth for this check, and
   644‚Üí  1944‚Üíevery animation component consults it before applying motion.
   645‚Üí  1945‚ÜíThe reduced-motion fallback is not just disabling animation; it is
   646‚Üí  1946‚Üídesigning the static end state to look intentional. A reveal
   647‚Üí  1947‚Üíanimation that fades from opacity 0 to opacity 1 must, in reduced-
   648‚Üí  1948‚Üímotion mode, render at opacity 1 immediately ‚Äî but the layout
   649‚Üí  1949‚Üíand spacing should still feel composed. A parallax that translates
   650‚Üí  1950‚Üían element must, in reduced-motion mode, render the element in
   651‚Üí  1951‚Üíits final position. The discipline of designing the static end state
   652‚Üí  1952‚Üíwell makes the reduced-motion experience equal in quality to the
   653‚Üí  1953‚Üíanimated experience, not a degraded version of it.

   657‚Üí  1957‚Üí6.7 Performance Budgets for Animation
   658‚Üí  1958‚ÜíAnimation performance is governed by two budgets: a frame
   659‚Üí  1959‚Üíbudget and a JavaScript budget. The frame budget is 16
   660‚Üí  1960‚Üímilliseconds per frame on a 60Hz display (8ms on 120Hz
   661‚Üí  1961‚Üídisplays); any animation that exceeds this budget causes visible
   662‚Üí  1962‚Üíjank. Aura Living animations only animate GPU-accelerated
   663‚Üí  1963‚Üíproperties (transform, opacity, filter) and never animate layout
   664‚Üí  1964‚Üíproperties (width, height, top, left, margin, padding). Will-change
   665‚Üí  1965‚Üíis used sparingly (on elements that will animate, set just before
   666‚Üí  1966‚Üíthe animation starts and removed just after) to hint the browser to
   667‚Üí  1967‚Üípromote the element to its own compositor layer.
   668‚Üí  1968‚ÜíThe JavaScript budget for animation libraries is 80KB gzipped
   669‚Üí  1969‚Üítotal: GSAP core plus ScrollTrigger plus @gsap/react plus Motion
   670‚Üí  1970‚Üíplus Lenis. This budget is enforced by a bundlesize check in CI; if
   671‚Üí  1971‚Üía PR introduces an animation library that pushes the total over
   672‚Üí  1972‚Üí80KB, the build fails. The current budget allocation is
   673‚Üí  1973‚Üíapproximately: GSAP core 30KB, ScrollTrigger 12KB, @gsap/react
   674‚Üí  1974‚Üí2KB, Motion 30KB, Lenis 6KB ‚Äî totaling 80KB. There is no
   675‚Üí  1975‚Üíheadroom; introducing any additional animation library requires
   676‚Üí  1976‚Üíremoving an existing one.
   677‚Üí  1977‚ÜíANTI-PATTERNS FORBIDDEN
   678‚Üí  1978‚ÜíThe following animation patterns are explicitly forbidden in
   679‚Üí  1979‚ÜíAura Living: (1) Animating layout properties (width, height, top,
   680‚Üí  1980‚Üíleft). (2) Using setTimeout or setInterval for animation. (3)
   681‚Üí  1981‚ÜíScroll event listeners without rAF throttling. (4) Parallax on
   682‚Üí  1982‚Üítouch devices without syncTouch testing. (5) Auto-playing video
   683‚Üí  1983‚Üíwith sound. (6) Animations longer than 1500ms. (7) More than
   684‚Üí  1984‚Üíthree simultaneous animations on screen at once. (8) Loading
   685‚Üí  1985‚Üíspinners that animate for less than 300ms (instant feedback
   686‚Üí  1986‚Üípreferred).
